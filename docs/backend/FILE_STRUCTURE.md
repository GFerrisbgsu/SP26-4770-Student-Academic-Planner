# Backend File Structure Guide

A comprehensive guide to every file in the Spring Boot backend for newcomers.

---

## 📁 Directory Overview

```
backend/
├── .github/                    # GitHub-specific files
├── .mvn/                       # Maven wrapper files
├── src/
│   ├── main/
│   │   ├── java/              # Java source code
│   │   └── resources/         # Configuration & migrations
│   └── test/                  # Test files
├── target/                     # Build output (auto-generated)
├── docker-compose.yml         # Docker services configuration
├── Dockerfile                 # Backend container build
├── pom.xml                    # Maven dependencies & build config
├── mvnw / mvnw.cmd            # Maven wrapper scripts
└── .gitignore                 # Git ignore rules
```

---

## 🔧 Root Configuration Files

### `pom.xml`
**Purpose:** Maven Project Object Model - defines dependencies and build configuration

**Key Sections:**
| Section | Purpose |
|---------|---------|
| `<parent>` | Inherits from Spring Boot 4.0.2 starter |
| `<properties>` | Sets Java version (25) |
| `<dependencies>` | Lists all project dependencies |
| `<build>` | Configures Maven plugins |

**Dependencies Included:**
- `spring-boot-starter-web` - REST API support
- `spring-boot-starter-data-jpa` - Database ORM
- `postgresql` - PostgreSQL driver
- `flyway-core` - Database migrations
- `h2` - In-memory database for testing

**When to modify:** Adding new dependencies (libraries)

---

### `docker-compose.yml`
**Purpose:** Defines Docker services for development environment

**Services Defined:**
| Service | Port | Purpose |
|---------|------|---------|
| `postgres` | 5432 | PostgreSQL database |
| `pgadmin` | 5050 | Database admin web UI |
| `backend` | 8080 | Spring Boot application |
| `frontend` | 3000 | React application |

**Key Configurations:**
- Database credentials: `appuser` / `apppass`
- Database name: `appdb`
- Volume: `smart_calendar_data` (persists data)
- Network: `sap-network` (connects all services)

**When to modify:** Adding new services, changing ports, updating environment variables

---

### `Dockerfile`
**Purpose:** Instructions for building the backend Docker image

**Build Stages:**
1. Uses Maven to compile Java code
2. Packages into executable JAR
3. Creates lightweight runtime image

**When to modify:** Changing Java version, build process, or runtime configuration

---

### `mvnw` / `mvnw.cmd`
**Purpose:** Maven Wrapper scripts (Unix/Windows)

**What they do:**
- Allow running Maven without installing it globally
- Ensure consistent Maven version across team
- Download Maven automatically if needed

**Usage:**
```powershell
./mvnw clean install      # Build project
./mvnw spring-boot:run    # Run application
./mvnw test               # Run tests
```

**When to modify:** Never (auto-generated)

---

### `.gitignore`
**Purpose:** Specifies files Git should ignore

**Ignored Items:**
- `target/` - Build output
- `*.class` - Compiled Java files
- `*.jar` - Package files
- `.idea/` - IntelliJ settings
- `*.log` - Log files

**When to modify:** Adding new file types to ignore

---

### `.gitattributes`
**Purpose:** Git configuration for file handling

**Configurations:**
- Line ending normalization
- Binary file detection

**When to modify:** Rarely needed

---

### `localhost.session.sql`
**Purpose:** SQL session file (auto-generated by database tools)

**When to modify:** Never (can be deleted)

---

## ☕ Java Source Code (`src/main/java/`)

All Java code is under the package: `com.sap.smart_academic_calendar`

### `Init.java`
**Location:** `com/sap/smart_academic_calendar/Init.java`

**Purpose:** Application entry point - starts Spring Boot

```java
@SpringBootApplication
public class Init {
    public static void main(String[] args) {
        SpringApplication.run(Init.class, args);
    }
}
```

**Annotations:**
- `@SpringBootApplication` - Enables auto-configuration, component scanning, and configuration

**When to modify:** Rarely - only for adding global configuration

---

## 📂 Config Package (`config/`)

Contains configuration classes for application-wide settings.

### `CorsConfig.java`
**Purpose:** Cross-Origin Resource Sharing configuration

**What it does:**
- Allows frontend (localhost:5173) to call backend API
- Enables all HTTP methods (GET, POST, PUT, DELETE)
- Allows credentials (cookies, auth headers)

**Allowed Origins:**
- `http://localhost:5173` - Vite dev server
- `http://localhost:3000` - Production frontend
- `http://frontend:3000` - Docker network

**When to modify:** Adding new frontend URLs, restricting methods

---

### `FlywayConfig.java`
**Purpose:** Database migration configuration

**What it does:**
- Manually configures Flyway for Spring Boot 4.x
- Runs SQL migrations on startup (dev profile only)
- Sets migration file location

**Active Profile:** Only runs with `@Profile("dev")`

**When to modify:** Changing migration settings, adding callbacks

---

## 🎮 Controller Package (`controller/`)

REST API endpoints - handles HTTP requests.

### `UserController.java`
**Purpose:** User management REST endpoints

**Endpoints:**
| Method | Path | Purpose |
|--------|------|---------|
| `GET` | `/api/users` | Get all users |
| `GET` | `/api/users/{id}` | Get user by ID |
| `POST` | `/api/users` | Create new user |

**Key Annotations:**
- `@RestController` - Marks as REST controller
- `@RequestMapping("/api/users")` - Base URL path
- `@GetMapping` / `@PostMapping` - HTTP method mapping
- `@PathVariable` - Extracts URL parameters
- `@RequestBody` - Parses JSON request body

**Design Pattern:**
- Contains NO business logic
- Delegates to `UserService`
- Returns `ResponseEntity` with appropriate status codes

**When to modify:** Adding new endpoints, changing URL paths

---

## 🔧 Service Package (`service/`)

Business logic layer - where the real work happens.

### `UserService.java`
**Purpose:** User business logic and validation

**Methods:**
| Method | Purpose |
|--------|---------|
| `getAllUsers()` | Retrieves all users as DTOs |
| `getUserById(Long id)` | Finds user by ID or throws exception |
| `createUser(CreateUserRequest)` | Validates and creates new user |

**Key Features:**
- Input validation (null checks, empty checks)
- Duplicate detection (username, email)
- Entity-to-DTO conversion
- Transaction management with `@Transactional`

**Key Annotations:**
- `@Service` - Marks as Spring service
- `@Transactional` - Database transaction on write operations

**When to modify:** Adding business rules, new operations

---

## 📦 Repository Package (`repository/`)

Data access layer - database queries.

### `UserRepository.java`
**Purpose:** User database operations

**Extends:** `JpaRepository<User, Long>`

**Auto-Generated Methods (from JpaRepository):**
- `findAll()` - Get all users
- `findById(Long id)` - Find by ID
- `save(User user)` - Save or update
- `deleteById(Long id)` - Delete by ID
- `count()` - Count records

**Custom Methods (Spring Data JPA generates SQL):**
| Method | Generated SQL |
|--------|---------------|
| `findByUsername(String)` | `WHERE username = ?` |
| `findByEmail(String)` | `WHERE email = ?` |
| `existsByUsername(String)` | `SELECT EXISTS(...WHERE username = ?)` |
| `existsByEmail(String)` | `SELECT EXISTS(...WHERE email = ?)` |

**When to modify:** Adding new query methods

---

## 📊 Model Package (`model/`)

JPA entities - Java classes mapped to database tables.

### `User.java`
**Purpose:** User entity - maps to `users` table

**Database Mapping:**
| Field | Column | Type | Constraints |
|-------|--------|------|-------------|
| `id` | `id` | BIGSERIAL | Primary Key, Auto-increment |
| `username` | `username` | VARCHAR(255) | NOT NULL, UNIQUE |
| `email` | `email` | VARCHAR(255) | NOT NULL, UNIQUE |
| `password` | `password` | VARCHAR(255) | NOT NULL |
| `createdAt` | `created_at` | TIMESTAMP | Auto-set on creation |

**Key Annotations:**
- `@Entity` - JPA entity marker
- `@Table(name = "users")` - Table name
- `@Id` - Primary key
- `@GeneratedValue(strategy = GenerationType.IDENTITY)` - Auto-increment
- `@Column` - Column configuration

**Constructors:**
- Default (required by JPA)
- Parametrized (for creating new users)

**When to modify:** Adding new fields, changing constraints

---

## 📤 DTO Package (`dto/`)

Data Transfer Objects - shapes data for API communication.

### `UserDTO.java`
**Purpose:** User response object (sent TO client)

**Fields:**
- `id` - User ID
- `username` - Username
- `email` - Email
- `createdAt` - Creation timestamp

**Note:** Password is intentionally EXCLUDED for security

**When to modify:** Changing API response shape

---

### `CreateUserRequest.java`
**Purpose:** User creation request (received FROM client)

**Fields:**
- `username` - Desired username
- `email` - Email address
- `password` - Password (validated in service)

**When to modify:** Changing required fields for user creation

---

## ⚙️ Resources (`src/main/resources/`)

Configuration files and static resources.

### `application.properties`
**Purpose:** Base configuration shared across all profiles

**Key Settings:**
- `spring.application.name` - App name
- `spring.profiles.active=local` - Default profile
- Database placeholders (overridden by profiles)
- Flyway settings
- Logging levels

**When to modify:** Adding global settings

---

### `application-local.properties`
**Purpose:** Local development with H2 in-memory database

**Features:**
- Uses H2 database (no Docker needed)
- Auto-creates schema on startup
- H2 console enabled at `/h2-console`
- Flyway disabled (uses JPA auto-DDL)

**When to use:** Quick testing without Docker

---

### `application-dev.properties`
**Purpose:** Development with PostgreSQL (Docker)

**Features:**
- Connects to PostgreSQL via environment variables
- Flyway migrations enabled
- SQL logging enabled

**When to use:** Full development with database persistence

---

### `application-docker.properties`
**Purpose:** Production-like Docker deployment

**Features:**
- Same as dev but for containerized deployment
- Uses Docker network hostnames

---

## 🔄 Adding New Features

### Adding a New Entity (e.g., Course)

1. **Create Migration:** `V3__create_courses.sql`
2. **Create Entity:** `model/Course.java`
3. **Create Repository:** `repository/CourseRepository.java`
4. **Create DTOs:** `dto/CourseDTO.java`, `dto/CreateCourseRequest.java`
5. **Create Service:** `service/CourseService.java`
6. **Create Controller:** `controller/CourseController.java`

### File Creation Order

```
1. Migration (SQL)     → Database table
2. Entity (Model)      → Java representation
3. Repository          → Data access
4. DTOs                → API contract
5. Service             → Business logic
6. Controller          → REST endpoints
```
